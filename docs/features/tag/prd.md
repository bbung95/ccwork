# 태그 기능 PRD

> 참조 스펙: `docs/features/tag/spec-fixed.md`
> 기술 결정 섹션은 다음 세션에서 채운다.

---

## 개요

노트에 태그를 자유롭게 붙이고 제거할 수 있다. 태그는 노트 편집 화면(NoteEditor)에서만 관리하며, 노트 저장 시 `title`, `content`와 함께 서버로 전송된다. 이번 범위에서는 태그 기반 필터/검색은 포함하지 않는다.

---

| ---- | ------ | ---------------------------------------------------------------- | ------------------------------------------------- |

| US-3 | 사용자 | 앞뒤 공백이 포함된 문자열을 입력하고 Enter를 누른다              | 공백이 자동으로 제거된 후 태그가 추가된다         |

## 기술 결정 (ADR)

> 결정 배경: 커스텀 훅 기반 합성 패턴을 통해 로직·UI·상태를 분리하고, 훅 단위 테스트가 가능한 구조를 우선한다.

### ADR-1: 태그 상태 관리 위치

- **Context**: NoteEditor는 이미 `title`, `content`를 로컬 state로 관리한다. 태그도 편집 중에만 의미 있는 임시 상태이며, Save 전까지 서버에 반영되지 않는다. 상태를 어느 레이어에 둘지 결정이 필요하다.
- **Decision**: `useTags(initialTags)` 커스텀 훅을 신규 생성하고, 태그 상태(`tags`)와 액션(`addTag`, `removeTag`, `setTags`)을 훅 안에 캡슐화한다. NoteEditor는 훅을 consume하고 `handleSave` 시 `tags`를 `updateNote` payload에 포함해 전달한다. NotesContext에는 태그 전용 액션을 추가하지 않는다.
- **Alternatives**:
  - 안 A — `tags`를 NoteEditor 로컬 state에 직접 추가 (`title`/`content`와 동일 레벨). 단순하지만 정규화·중복 검사 로직이 컴포넌트에 혼재된다.
  - 안 C — NotesContext에 `addTag(noteId, tag)` / `removeTag(noteId, tag)` 액션 추가. 즉시 저장을 해야 할 때만 적합하며, 현재 스펙(Save 시 일괄 저장)과 맞지 않는다.
- **Consequences**:
  - (+) `useTags`만 import해 렌더 없이 단위 테스트 가능.
  - (+) 정규화·중복 검사 로직이 훅 한 곳에 집중되어 NoteEditor 파일 길이 감소.
  - (+) 동일 패턴을 다른 입력 로직에도 적용하기 쉽다.
  - (-) 파일 4개 추가(`src/hooks/useTags.ts`, `src/components/TagChipBase.tsx`, `src/components/EditorTagChip.tsx`, `src/components/TagInput.tsx`) — 현재 규모에서는 소폭의 오버헤드.
  - (-) `useTags(initialTags)`의 초기값은 마운트 시 한 번만 적용된다. NoteEditor의 `useEffect`에서 노트 전환 시 `setTags(selectedNote?.tags ?? [])`를 명시적으로 호출해야 하며, 이를 빠뜨리면 이전 노트의 태그가 잔류하는 버그가 발생한다.

---

### ADR-2: 소문자 정규화 시점

- **Context**: 입력값을 정규화하는 시점은 여러 곳이 될 수 있다 — 입력 중(onChange), 추가 확정 시(addTag), 저장 시(handleSave), API 레이어. 시점에 따라 UX와 책임 위치가 달라진다.
- **Decision**: `useTags` 내부 `addTag` 호출 시점에 `trim()` → `toLowerCase()` 를 적용한다. 입력 필드(`<input>`)는 사용자가 타이핑하는 원본 값을 그대로 표시하고, Enter를 누르는 순간에만 정규화한다.
- **Alternatives**:
  - onChange 시점 정규화 — 입력 중 글자가 자동으로 소문자로 바뀌어 사용자 혼란을 준다.
  - handleSave 시점 정규화 — 정규화 로직이 useTags 밖으로 빠져나가 중복 검사 기준이 분산된다.
  - API 레이어 정규화 — 클라이언트 중복 검사가 불가능해지고 서버 의존성이 생긴다.
- **Consequences**:
  - (+) 중복 검사(`tags.includes(tag)`)도 같은 지점에서 수행하므로 `React`와 `react`가 일관되게 처리된다.
  - (+) 정규화 책임이 `useTags` 안에만 있어 테스트 경계가 명확하다.
  - (-) 입력 필드에는 원본 대소문자가 보이다가 chip 생성 시 소문자로 바뀌는 시각적 전환이 발생한다. 사용자 입장에서 입력한 것과 결과가 달라 보이는 이질감이 생길 수 있다.

### ADR-3: 태그 입력 UX (Enter vs 버튼)

- **Context**: 태그 추가 트리거를 어떻게 제공할지 결정이 필요하다. 버튼 추가 방식, Enter 키 방식, 구분자(`,`) 방식 등 여러 옵션이 있다.
- **Decision**: **Enter 키 단독**으로 태그를 추가한다. `onKeyDown` 핸들러에서 `key === 'Enter'`일 때 `addTag`를 호출하고 입력 필드를 초기화한다. 별도 추가 버튼은 두지 않는다.
- **Alternatives**:
  - Enter + 추가(+) 버튼 병행 — 모바일 접근성에는 유리하나 UI가 복잡해진다.
  - 쉼표(`,`) 구분자로 여러 태그 일괄 입력 — 쉼표 포함 태그명을 허용할 수 없게 되고 스펙 범위를 벗어난다.
  - onBlur 자동 추가 — 태그를 추가할 의도 없이 필드를 떠났을 때도 추가되어 의도치 않은 동작이 발생한다.
- **Consequences**:
  - (+) 추가 버튼이 없어 태그 영역 레이아웃이 단순하다.
  - (+) `onKeyDown` 핸들러가 유일한 추가 진입점이므로 테스트 경로가 단순하다.
  - (-) 모바일 소프트키보드에서는 Enter 키가 줄바꿈으로 동작하거나 접근이 불편한 경우가 있어 태그 추가가 어렵다. 현재 Out of Scope이나 모바일 지원 시 추가 버튼 또는 구분자 방식으로 재검토가 필요하다.

### ADR-4: 기존 노트 마이그레이션 전략 (`tags` 필드 부재)

- **Context**: 현재 `db.json`에 저장된 기존 노트들에는 `tags` 필드가 없다. `Note` 타입에 `tags: string[]`을 추가하면 기존 데이터와 타입이 불일치한다.
- **Decision**: **클라이언트 방어적 읽기**로 처리한다. `Note` 타입은 `tags: string[]` (required)로 유지하고, NoteEditor의 `useEffect` 동기화 시 `selectedNote.tags ?? []`로 기본값을 주입한다. `db.json` 수정이나 서버 마이그레이션은 수행하지 않는다.
- **Alternatives**:
  - `db.json` 수동 편집 — 기존 노트 전체에 `"tags": []` 추가. 확실하지만 데이터 파일을 직접 건드려야 하고 향후 노트 추가 시 누락 가능.
  - TypeScript 타입을 `tags?: string[]` (optional)로 변경 — 코드 전반에 `tags?.length` 방어 코드가 퍼지고, 저장 시 undefined를 보내는 버그 위험이 있다.
  - fetchNotes 응답을 normalize하는 함수 추가 — API 레이어에서 `tags: note.tags ?? []`로 보정. 타입 일관성은 가장 강하지만 현재 규모에서 과도하다.
- **Consequences**:
  - (+) `db.json` 변경 없이 기존 노트를 그대로 열 수 있다.
  - (+) 타입은 `required`로 유지되므로 코드 전반에 `?.` 방어 코드가 불필요하다.
  - (+) 기존 노트를 열어 Save 하면 그 시점부터 `tags: []`가 서버에 기록되어 자연스럽게 마이그레이션된다.
  - (-) Save 하지 않은 기존 노트는 서버에 `tags` 필드가 없는 상태로 남는다. `?? []` 처리를 빠뜨린 접근 경로가 생기면 런타임 오류 위험이 계속 존재한다.
  - (-) 타입이 `required`임에도 실제 데이터가 필드를 보장하지 않는 불일치가 Save 전까지 유지된다. TypeScript 컴파일러가 이 위험을 잡아주지 못한다.

## Out of Scope

이번 구현 범위에 명시적으로 포함되지 않는 기능들이다.

### 태그 탐색 / 검색

- 태그를 기준으로 노트를 필터링하는 기능
- 태그 클릭 시 동일 태그를 가진 노트 목록으로 이동하는 기능
- 전역 태그 목록(태그 클라우드, 태그 사이드바) 표시
- 태그 자동완성 (기존 태그 목록 기반 suggestion)

### NoteList 표시

- NoteItem 카드에 태그 chip을 표시하는 기능
- 태그 유무 또는 특정 태그 포함 여부에 따른 카드 시각적 구분

### 태그 편집 / 일괄 관리

- 기존 태그의 이름을 수정하는 기능 (삭제 후 재입력으로 대체)
- 특정 태그를 전체 노트에서 일괄 삭제 또는 이름 변경
- 태그 색상 지정

### 태그 정렬 / 순서

- 추가된 태그를 알파벳순 또는 사용자 정의 순서로 정렬
- 드래그앤드롭으로 태그 순서 변경

### 서버 사이드 태그 관리

- 독립적인 `tags` 리소스 엔드포인트 (`/tags`)
- 태그 사용 빈도나 연결된 노트 수 집계
- 태그 기반 서버 사이드 필터 쿼리 (`/notes?tags=react`)

### 입력 제한 외 유효성 검사

- 특수문자 입력 제한 (현재: 모든 문자 허용, 20자 상한만 존재)
- 태그 최소 글자 수 제한

### 자동 저장

- 태그 변경 시 즉시 자동 저장 (현재: Save 버튼 클릭 시 일괄 저장)

## 용어 정의

| 용어                      | 정의                                                                                                                                           |
| ------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| **태그(Tag)**             | 노트에 붙이는 짧은 문자열 레이블. 소문자로 정규화되어 저장된다.                                                                                |
| **Chip**                  | 태그 하나를 시각적으로 표현하는 UI 요소. `EditorTagChip` 컴포넌트로 구현하며, 텍스트와 X(삭제) 버튼을 포함한다. `TagChipBase`를 기반으로 한다. |
| **TagChipBase**           | 칩 외형(shape, color, spacing)만 담당하는 공유 원시 컴포넌트. `EditorTagChip`과 `FilterTagChip`이 내부적으로 사용한다. 직접 렌더하지 않는다.   |
| **정규화(Normalization)** | 입력값을 `trim()` 후 `toLowerCase()`를 적용해 저장 형식으로 변환하는 과정.                                                                     |
| **중복 태그**             | 정규화 결과가 이미 존재하는 태그와 일치하는 입력. 무시된다.                                                                                    |
| **태그 영역**             | NoteEditor에서 제목 입력 필드와 본문 입력 필드 사이에 위치하는 UI 블록. 태그가 없으면 렌더링하지 않는다.                                       |
| **비활성화 상태**         | 태그가 10개에 도달했을 때 입력 필드가 `disabled` 처리되는 상태.                                                                                |
| **저장 시점**             | Save 버튼 클릭 시 `title`, `content`, `tags`를 서버에 함께 전송하는 단일 시점.                                                                 |
