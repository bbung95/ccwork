# 필터 기능 PRD

> 참조 스펙: `docs/features/filter/spec-fixed.md`

---

## 개요

태그를 기준으로 노트 목록을 필터링할 수 있다. 필터 UI는 `NoteList` 상단에 태그 칩(chip) 형태로 배치되며, 다중 선택 시 OR 조건으로 동작한다. 필터 상태는 세션 동안 유지되고 새로고침 시 초기화된다.

---

## 사용자 스토리

| #     | 역할   | 행동                                           | 기대 결과                                                                 |
| ----- | ------ | ---------------------------------------------- | ------------------------------------------------------------------------- |
| US-1  | 사용자 | NoteList 상단의 태그 칩 목록을 본다            | 전체 노트에 존재하는 태그 합집합이 칩 형태로 표시된다                     |
| US-2  | 사용자 | 태그 칩을 클릭한다                             | 해당 태그가 선택되고 OR 조건으로 노트 목록이 즉시 필터링된다              |

| US-5  | 사용자 | "필터 초기화" 버튼을 클릭한다                  | 모든 태그 선택이 해제되고 전체 노트가 표시된다                            |
| US-6  | 사용자 | 아무 태그도 선택하지 않은 상태이다             | 전체 노트가 표시된다                                                      |
| US-7  | 사용자 | 필터 조건에 맞는 노트가 없다                   | "선택한 태그와 일치하는 노트가 없습니다." 빈 상태 메시지가 표시된다       |
| US-8  | 사용자 | 필터가 선택된 상태에서 다른 노트를 클릭한다    | 필터 상태가 유지된다                                                      |
| US-9  | 사용자 | 필터가 선택된 상태에서 페이지를 새로고침한다   | 필터가 초기화되어 전체 노트가 표시된다                                    |
| US-10 | 사용자 | 필터 중인 상태에서 새 노트를 생성한다          | 필터 상태는 변경되지 않으며, 새 노트가 조건 미충족 시 목록에 미표시된다   |
| US-11 | 사용자 | 전체 노트에 태그가 하나도 없다                 | TagFilter 영역 자체가 렌더링되지 않는다                                   |
| US-12 | 사용자 | 노트를 삭제해 해당 태그를 가진 노트가 없어진다 | 태그 칩 목록에서 해당 태그가 자동으로 사라진다                            |

## 기술 결정 (ADR)

> 결정 배경: 커스텀 훅 기반 합성 패턴을 따라 필터 로직을 분리하고, 훅 단위 테스트가 가능한 구조를 우선한다. (태그 기능 ADR-1과 동일한 원칙)

### ADR-1: 필터 상태 관리 위치

- **Context**: `selectedTags` 상태를 어느 레이어에서 관리할지 결정이 필요하다. 스펙은 `App.tsx` 또는 `NotesContext`를 예시로 제시하나, 현재 아키텍처에서 필터 상태는 NoteList 표시에만 영향을 준다. `selectedNoteId`(App.tsx)처럼 상위로 올릴지, 아니면 NoteList 레벨에서 캡슐화할지 선택이 필요하다.
- **Decision**: `useFilter(notes)` 커스텀 훅을 신규 생성하고 `NoteList`에서 consume한다. 훅이 `selectedTags`, `filteredNotes`, `allTags`, `toggleTag`, `clearFilter`를 반환한다. 필터 상태는 NoteList 로컬 범위로 유지하며 App.tsx나 NotesContext로 올리지 않는다.
- **Alternatives**:
  - App.tsx에 `selectedTags` 추가 — `selectedNoteId`와 동일 레벨 관리. 향후 NoteEditor나 다른 컴포넌트에서 필터 상태를 읽어야 할 때 이미 올라가 있어 편리하지만, 현재는 NoteList 외에 소비처가 없다.
  - NotesContext에 포함 — 필터 로직과 노트 데이터 로직이 혼재되어 단일 책임 원칙에서 벗어난다. 필터는 서버 상태가 아닌 순수 클라이언트 UI 상태다.
- **Consequences**:
  - (+) `useFilter`만 import해 렌더 없이 단위 테스트 가능.
  - (+) `filteredNotes`, `allTags` 파생 계산이 훅 안에 캡슐화되어 NoteList가 단순해진다.
  - (+) 태그 기능의 `useTags`와 동일한 패턴으로 일관성이 생긴다.
  - (-) 향후 NoteEditor나 헤더 등 다른 컴포넌트에서 필터 상태를 읽어야 할 경우 App.tsx로 끌어올리는 리팩터가 필요하다.
  - (-) 파일 3개 추가(`src/hooks/useFilter.ts`, `src/components/FilterTagChip.tsx`, `src/components/TagFilter.tsx`). `TagChipBase.tsx`는 태그 기능에서 이미 추가됨.

### ADR-2: 다중 선택 필터 조건 (OR vs AND)

- **Context**: 태그를 2개 이상 선택했을 때 두 태그를 모두 가진 노트만 표시할지(AND), 하나라도 가진 노트를 표시할지(OR) 결정이 필요하다.
- **Decision**: **OR 조건**. `note.tags.some(tag => selectedTags.includes(tag))`로 구현한다.
- **Alternatives**:
  - AND 조건 — `note.tags` 가 `selectedTags`를 모두 포함해야 표시. 태그를 많이 선택할수록 결과가 급격히 줄어들어 사용성이 떨어진다.
  - 단일 선택만 허용 — 다중 선택 자체를 막음. 가장 단순하지만 스펙 요구사항에서 벗어난다.
- **Consequences**:
  - (+) 선택 태그가 늘어날수록 결과가 늘어나므로 사용자가 원하는 노트를 찾기 쉽다.
  - (+) 필터링 로직이 단순(`some`)해 테스트 케이스 작성이 쉽다.
  - (-) 태그를 많이 선택하면 결과가 전체 목록과 거의 같아지는 시점이 생겨 필터의 효용이 희석된다.
  - (-) 사용자가 AND/OR 동작을 직관적으로 알기 어려울 수 있다 (UI에서 동작 방식을 명확히 전달해야 함).

### ADR-3: TagFilter에 표시할 태그 목록 범위

- **Context**: 필터 칩 목록에 어떤 태그를 표시할지 결정이 필요하다. 현재 필터 결과 기준으로 줄일지, 전체 노트 기준으로 고정할지에 따라 UX가 달라진다.
- **Decision**: **전체 노트의 태그 합집합**을 표시한다. `useFilter` 내부에서 `notes.flatMap(n => n.tags)`의 중복을 제거해 `allTags`를 파생한다. 현재 필터 결과와 무관하게 목록이 일정하다.
- **Alternatives**:
  - 현재 필터 결과 기준 — 선택한 태그에 따라 남은 칩이 줄어드는 방식. 선택 순서에 따라 선택지가 사라져 혼란스럽고, 이전에 선택한 태그를 해제하기 위해 찾을 수 없게 되는 문제가 발생한다.
  - 수동 태그 목록 관리 (`/tags` 엔드포인트) — 서버 의존성이 생기고 Out of Scope.
- **Consequences**:
  - (+) 어떤 태그를 선택하든 칩 목록이 변하지 않아 예측 가능한 UX.
  - (+) `allTags` 계산이 `notes`에서 파생되므로 노트 삭제 시 자동으로 태그 칩도 사라진다.
  - (-) 필터를 적용해도 칩 목록이 줄지 않아 시각적으로 "선택의 여지가 있는지" 파악하기 어렵다.
  - (-) 노트가 많을수록 `flatMap` + 중복 제거 연산 비용이 선형으로 증가한다. 현재 규모에서는 무시할 수준이지만 노트 수가 수천 건을 넘으면 메모이제이션(`useMemo`)이 필요하다.

### ADR-4: 필터 상태 유지 범위

- **Context**: 사용자가 필터를 설정한 후 페이지를 이탈하거나 새로고침했을 때 상태를 어디까지 유지할지 결정이 필요하다.
- **Decision**: **세션 내 메모리 유지, 새로고침 시 초기화**. React 상태로만 관리하며 `localStorage`나 URL 파라미터에 저장하지 않는다.
- **Alternatives**:
  - localStorage 저장 — 새로고침 후에도 필터가 유지됨. 동기화 로직과 초기화 처리가 추가로 필요하며, 앱 규모 대비 오버엔지니어링이다.
  - URL 쿼리 파라미터 — 필터 상태를 URL로 공유 가능. 라우터 의존성이 생기고 현재 프로젝트에는 라우터가 없다.
- **Consequences**:
  - (+) 구현이 단순하다. React 상태만으로 동작하고 외부 저장소 의존성이 없다.
  - (+) 앱 진입 시 항상 전체 보기 상태로 시작해 예측 가능한 초기 상태가 보장된다.
  - (-) 새로고침하면 필터가 사라져 특정 태그를 반복적으로 보는 사용자는 매번 재선택해야 한다.
  - (-) 세션 유지라는 동작이 명시적으로 드러나지 않아 사용자가 새로고침 후 필터가 사라진 것을 버그로 오해할 수 있다.

---

### ADR-5: 필터 칩 컴포넌트 구조

- **Context**: 필터 칩(`FilterTagChip`)과 에디터 칩(`EditorTagChip`)은 외형이 거의 동일하지만 동작이 다르다. 단일 컴포넌트로 합칠지, 별도 컴포넌트로 분리할지 결정이 필요하다.
- **Decision**: **`TagChipBase` + 특화 컴포넌트** 패턴을 사용한다. `TagChipBase`가 외형(shape, color, spacing)만 담당하고, `FilterTagChip`과 `EditorTagChip`이 각자의 동작을 추가한다. 스펙: `docs/design-system/components/TagChip.md` 참조.
- **Alternatives**:
  - 단일 `TagChip` + variant prop — 처음엔 단순하지만 `onRemove`, `selected`, `onClick`, `disabled` 등 props가 기능 추가마다 누적되어 내부 조건문이 복잡해진다.
  - 완전 독립된 두 컴포넌트 — 기본 외형 변경 시 두 파일을 함께 수정해야 해 동기화 오류 위험이 생긴다.
- **Consequences**:
  - (+) 새 칩 변형은 `TagChipBase`를 기반으로 파일만 추가하면 되어 기존 컴포넌트를 건드리지 않는다.
  - (+) 각 컴포넌트의 props가 자신의 역할에만 집중해 타입이 명확하다.
  - (-) `TagChipBase`가 태그 기능 구현 선행 필요 — 필터 기능 단독 구현 시 `TagChipBase`를 먼저 만들어야 한다.

## Out of Scope

이번 구현 범위에 명시적으로 포함되지 않는 기능들이다.

### 필터 조건 확장

- AND 조건 필터링 (현재: OR 조건 고정)
- 태그 외 필터 기준: 생성일·수정일 범위, 핀 상태, 제목·내용 키워드
- 복합 조건 조합 (예: "태그 A AND 날짜 B 이후")

### 필터 상태 영구 저장

- `localStorage`에 `selectedTags` 저장 (새로고침 후 복원)
- URL 쿼리 파라미터 연동 (`?tags=react,typescript`) — 공유·북마크 지원
- 사용자 정의 필터 프리셋 저장 및 불러오기

### 태그 칩 정렬 및 그룹화

- 태그 칩을 알파벳 순, 사용 빈도 순으로 정렬
- 사용 중인 태그와 미사용 태그를 시각적으로 구분
- 태그 칩 목록이 많을 때 접기/펼치기(collapse)

### 서버 사이드 필터링

- `/notes?tags=react` 형태의 서버 필터 쿼리 (현재: 클라이언트 필터링)
- 태그 사용 빈도 집계 API

### NoteList 외부 필터 접근

- 헤더, NoteEditor, 모달 등에서 현재 필터 상태를 읽거나 변경
- 필터 상태를 기반으로 NoteEditor에서 태그 자동 입력

### 빈 상태 액션

- 필터 빈 상태 화면에서 "필터 초기화" 버튼 노출 (현재: NoteList 상단 버튼만)
- 빈 상태에서 새 노트 생성 CTA

## 용어 정의

| 용어              | 정의                                                                                                                                                        |
| ----------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **TagFilter**     | NoteList 상단에 위치하는 태그 칩 목록 컴포넌트. 선택/해제 토글과 초기화 버튼을 포함한다.                                                                    |
| **TagChip**       | 필터에서 개별 태그를 표현하는 UI 요소. `FilterTagChip` 컴포넌트로 구현하며, 클릭으로 선택/해제 토글된다. `TagChipBase`(태그 기능에서 도입)를 기반으로 한다. |
| **selectedTags**  | 현재 선택된 태그 배열. 빈 배열(`[]`)이면 전체 보기 상태.                                                                                                    |
| **allTags**       | 전체 노트에서 수집한 태그 합집합. `TagFilter`에 표시되는 칩 목록의 원본.                                                                                    |
| **filteredNotes** | `selectedTags`를 OR 조건으로 적용해 도출한 노트 배열. `selectedTags`가 비어 있으면 전체 노트.                                                               |
| **OR 조건**       | 선택된 태그 중 하나라도 포함한 노트를 결과에 포함하는 필터 방식.                                                                                            |
| **필터 초기화**   | `selectedTags`를 빈 배열로 되돌리는 동작. 2개 이상 선택 시 노출되는 버튼으로 트리거한다.                                                                    |
| **세션 유지**     | 페이지 새로고침 전까지 React 상태로만 필터를 보존하는 방식. 새로고침 시 초기화된다.                                                                         |
